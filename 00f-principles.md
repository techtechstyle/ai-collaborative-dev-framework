## 階層構造がもたらすもの [L2]

### 競合が発生しない理由

従来の「4つの哲学が並列」という捉え方では、以下のような「競合」が発生していました:

| 状況 | 従来の捉え方(競合) |
|-----|-------------------|
| 検証を増やすか減らすか | 急がば回れ vs シンプルさ |
| ドキュメントを書くか省略するか | 持続可能性 vs シンプルさ |
| 損切りを強制するか待つか | 急がば回れ vs 心理的安全性 |

階層構造では、これらは「競合」ではなく「上位の制約内での最適化」になります:

| 状況 | 階層での捉え方 |
|-----|---------------|
| 検証を増やすか減らすか | L0-L2を満たす範囲で、L3を最適化 |
| ドキュメントを書くか省略するか | L0に寄与するなら書く。しないなら省略可 |
| 損切りを強制するか待つか | L1を損なわない形で、L2を適用 |

### 判断の一貫性

階層構造により、判断が一貫します:

```
┌─────────────────────────────────────────────────────────────────┐
│                    階層判断フロー                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  判断が必要な状況                                               │
│          │                                                      │
│          ↓                                                      │
│  L0 チェック: 持続可能性に寄与するか?                         │
│          │       「1年後のチームにプラスか?」                  │
│          │                                                      │
│     No ──┼─→ その選択肢は却下                                  │
│          │                                                      │
│          ↓ Yes                                                  │
│  L1 チェック: 心理的安全性を損なわないか?                     │
│          │       「誰かが萎縮しないか?」                       │
│          │                                                      │
│     No ──┼─→ 萎縮しない方法を検討                              │
│          │                                                      │
│          ↓ Yes                                                  │
│  L2 チェック: 急がば回れを実践できるか?                       │
│          │       「丁寧さを保てるか?」                         │
│          │                                                      │
│     No ──┼─→ 丁寧にできる範囲に縮小                            │
│          │                                                      │
│          ↓ Yes                                                  │
│  L3 チェック: シンプルさの範囲内か?                           │
│          │       「1文1検証を満たせるか?」                     │
│          │                                                      │
│     No ──┼─→ よりシンプルな方法を選択                          │
│          │                                                      │
│          ↓ Yes                                                  │
│  L4 チェック: AIの強みを活用できるか? ★新設★                 │
│          │       「AIが得意な部分はあるか?」                   │
│          │                                                      │
│     No ──┼─→ 人間主導で進める（通常フロー）                    │
│          │                                                      │
│          ↓ Yes                                                  │
│  — AIファースト実践（F1-F4適用）                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 具体例:階層で判断する

#### 例1: ドキュメントをどこまで書くか

```
状況: 実装が終わったが、ドキュメントを書く時間がない

階層での判断:
 L0 チェック: ドキュメントは1年後のチームにプラスか? → Yes
 L3 適用: L0を満たした上で、シンプルに書けるか? → シンプルに書く

結論: 最低限(CLAUDE.mdの更新、要点のみのコメント)は残す
      詳細ドキュメントは「後で書く」とTODOに残す
```

#### 例2: 検証をどこまでやるか

```
状況: 小さな修正だが、フルテストを回すと30分かかる

階層での判断:
 L2 チェック: 丁寧さを保てるか? → 状況による
  → 本番リリース前: L2優先(フルテスト)
  → 開発中の小修正: L3適用(関連テストのみ)
 判断基準: 「失敗した場合の影響範囲」
```

#### 例3: 損切りを強制するか

```
状況: 新人が1時間格闘しているが、本人は「もう少しで解けそう」と言っている

階層での判断:
 L1 チェック: 誰かが萎縮しないか? → 強制は萎縮を招く恐れ
 L2 チェック: 丁寧さを保てるか? → 損切りラインは超えている

両立させる方法:
 → 「素晴らしい粘り強さだね。ここで一度整理しよう」と声をかける
 → 一緒に問題を分解し、より小さな単位で再挑戦させる
 → 「撤退」ではなく「戦略的再構成」としてフレーミング
```

## 階層から導かれる15の原則 [L2]

4つの哲学は抽象的な「考え方」です。
これを日々の実践に落とし込むために、各Levelごとに原則を定義します。

```
┌─────────────────────────────────────────────────────────────────┐
│                    階層 → 原則 → 実践                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Level(Why)       なぜそうするのか                            │
│       │                                                         │
│       ↓                                                         │
│  原則(What)       何を守るべきか   ← このセクション           │
│       │                                                         │
│       ↓                                                         │
│  実践(How)        どう実行するか   ← Part 1-4                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Level 0【目的】持続可能性の原則(G1-G3, R1-R3)

| # | 原則 | 説明 | 判断基準 |
|---|------|------|---------|
| **G1** | **任せるだけでなく学ぶ** | AIの出力を理解し、自分のスキルも向上させる | このコードを自分で説明できるか? |
| **G2** | **個人 → チームへ展開** | 個人の知見を形式知化し、チームで共有する | 学びを記録しているか? |
| **G3** | **CLAUDE.mdは監視の憲法** | CLAUDE.mdは「人間の監視をスケールさせる仕組み」として育てる | 1年後も価値があるか? |
| **R1** | **スキルの可逆性** | 人間がAIなしでも作業できる能力を維持する | このコード、自分で書けるか? |
| **R2** | **プロセスの可逆性** | AIを外してもワークフローが成立する設計にする | AIなしでも手順は成立するか? |
| **R3** | **判断の可逆性** | AIの提案をいつでも却下・覆せる状態を保つ | この判断を覆す根拠を説明できるか? |

#### G3の深掘り:CLAUDE.mdは「監視の憲法」

CLAUDE.mdは単なる「Claudeへの指示書」ではありません。
**人間の監視をスケールさせる仕組み**です。

```
┌─────────────────────────────────────────────────────────────────┐
│              CLAUDE.md = Scalable Oversight                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  人間が毎回監視しなくても、原則が守られる仕組み                 │
│                                                                 │
│  従来の監視:                                                    │
│  ────────────                                                   │
│  人間 → [毎回確認] → Claudeの出力                               │
│  問題: 人間の時間は有限、すべてを確認できない                   │
│                                                                 │
│  CLAUDE.mdによる監視:                                           │
│  ────────────────────                                           │
│  人間 → [原則を定義] → CLAUDE.md → [原則を適用] → Claudeの出力  │
│  効果: 人間が一度定義した原則が、すべてのセッションに適用       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**CLAUDE.mdに書くべきこと:**

| カテゴリ | 内容 | 効果 |
|---------|------|------|
| **絶対的禁止事項** | 「〇〇は絶対にしない」 | 人間が毎回確認しなくても違反を防げる |
| **判断基準** | 「迷ったら〇〇を優先」 | 人間の判断基準をClaudeが代行 |
| **品質基準** | 「〇〇を満たさないコードは提出しない」 | 人間のレビュー負荷を軽減 |
| **学習記録** | 「過去に〇〇で失敗した」 | 同じ失敗を人間が監視しなくても防げる |

> **参考**: Anthropicは「少量の高品質な人間の監視を、AIの支援によって大規模に拡張する」
> ことを「Scalable Oversight」と呼んでいます。CLAUDE.mdはこの概念のプロジェクト実装です。

### AI行動原則（A1-A4）：アウトプットの責任

AIがアウトプット時に守るべき原則です。人間の原則と対応関係があります。

## このガイドが向いている人・向いていない人 [L1]

### 向いている人

このガイドは、以下のような考えを持つ人に向いています。

- **「速く」より「確実に」を重視したい**
  - 手戻りのない開発をしたい
  - 品質を犠牲にしたくない
  - 長期的な視点で考えたい

- **失敗から学ぶ姿勢がある**
  - 失敗を隠さず、記録したい
  - 同じ失敗を繰り返したくない
  - 継続的に改善したい

- **AIを「道具」ではなく「パートナー」と捉えたい**
  - AIの力を引き出すのは人間の責任と考えられる
  - うまくいかないとき、自分の側も振り返れる
  - 人間としてのスキルも磨きたい

- **複雑さと戦う覚悟がある**
  - シンプルにするための努力を惜しまない
  - 分解・整理・言語化を大切にしたい
  - 「面倒」を避けずに向き合いたい

### 向いていない人

一方、以下のような考えを持つ人には、このガイドは向いていないかもしれません。

- **「とにかく速く」を最優先したい**
  - 品質より速度を重視する
  - 手戻りは後で考える
  - 今日の締め切りが最重要

- **失敗を記録するのは面倒**
  - 動けばいい
  - 振り返りは時間の無駄
  - 同じ失敗をしてもその時対処すればいい

- **AIは効率化の道具**
  - うまくいかないのはAIのせい
  - AIに任せられることは全部任せたい
  - 自分で理解する必要はない

- **複雑なままでいい**
  - 分解は面倒
  - 動けばいい
  - シンプルさに価値を感じない

**どちらが正しいということではありません。**
価値観の違いです。

このガイドの哲学に共感できない場合は、別のアプローチを探すことをお勧めします。
世の中には様々なAI協働の方法論があります。
自分に合ったものを見つけることが大切です。
