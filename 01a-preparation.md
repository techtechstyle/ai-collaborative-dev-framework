## 1. 準備量の考え方 [L2]

### 核心:目的によって適切な準備量は異なる

| 目的 | 準備量 | 理由 |
|------|--------|------|
| ワークフロー習得・学習 | 最小限 | 準備に時間を取られると本題に入れない |
| 個人の小規模プロジェクト | 軽め | 頭の中で把握できる範囲なら文書化不要 |
| チーム開発・本番システム | しっかり | 認識合わせと後からの参照に必要 |
| AI協働開発 | 中間〜しっかり | AIのコンテキスト提供に必要 |

### 従来の考え方との違い

```
┌─────────────────────────────────────────────────────────────────┐
│  従来のシステム開発                                              │
│                                                                 │
│    設計変更コスト  >>>  実装コスト                            │
│                                                                 │
│    → 設計を完璧にする価値がある                                  │
│    → 手戻りは大きなロス                                         │
├─────────────────────────────────────────────────────────────────┤
│  AI協働開発                                                      │
│                                                                 │
│    設計変更コスト  ≒  実装コスト(どちらも低い)                 │
│                                                                 │
│    → 試行錯誤しながら洗練させる方が効率的                        │
│    → 早期の検証で軌道修正                                        │
└─────────────────────────────────────────────────────────────────┘
```

### 「完璧な準備」から「十分な準備」へ

| 項目 | 事前に準備 | 進化させる |
|------|-----------|-----------|
| プロジェクトの目的 | ✅ 明確に | - |
| 主要な要件 | ✅ 概要レベル | 詳細は後から |
| 技術スタック | ✅ 大枠を決定 | 微調整あり |
| アーキテクチャ方針 | ✅ 基本方針 | 詳細は実装で |
| CLAUDE.md | △ 最小限 | ✅ 育てる |
| コーディング規約 | △ 基本のみ | ✅ 経験から追加 |
| 詳細設計 | - | ✅ 実装しながら |

## 2. 一般的ベストプラクティスとの関係 [L2]

### 大部分は一致する

良い設計の本質は「人間が理解しやすい」こと。
Claudeは人間が書いたコード・文書から学習しているため、
人間に優しい設計はClaudeにも優しい。

> 💡 「80%/20%」という比率は、パレートの法則(80:20の法則)を設計原則に適用した**経験則**です。
> 正確な比率ではなく、「大部分は共通、一部がClaude特有」という考え方を示しています。

```
┌─────────────────────────────────────────────────────────────────┐
│                      設計原則の関係                              │
│                                                                 │
│         ┌─────────────────────────────┐                         │
│         │      良い設計の原則         │                         │
│         │   (人間が理解しやすい)     │                         │
│         │                             │                         │
│         │  ┌───────────────────────┐  │                         │
│         │  │   大部分: 共通領域    │  │                         │
│         │  │   ・単一責任          │  │                         │
│         │  │   ・疎結合            │  │                         │
│         │  │   ・明示的な依存      │  │                         │
│         │  │   ・小さな関数        │  │                         │
│         │  │   ・型による契約      │  │                         │
│         │  └───────────────────────┘  │                         │
│         │                             │                         │
│         │  ┌───────────┐              │                         │
│         │  │ 一部:    │ Claude特有の最適化                      │
│         │  │ 物理分割 │ ファイルサイズ等                        │
│         │  └───────────┘              │                         │
│         └─────────────────────────────┘                         │
└─────────────────────────────────────────────────────────────────┘
```

### 一致する原則

| 原則 | 一般的な理由 | Claude的な理由 |
|------|-------------|---------------|
| 単一責任 | 変更理由を1つに | 1タスク = 1指示で完結 |
| 疎結合 | 変更影響を局所化 | 必要なファイルだけ参照 |
| 明示的な依存 | テスト容易性 | 暗黙の前提を減らす |
| 小さな関数 | 可読性 | コンテキストに収まる |
| 型による契約 | バグ防止 | 意図が明確に伝わる |

### Claude特有の最適化

| 観点 | 一般的最適化 | Claude最適化 |
|------|-------------|-------------|
| ファイルサイズ | 論理単位でまとめる | 物理的に小さく(目安200-300行) |
| 命名規則 | チーム内共有 | CLAUDE.mdに明記 |
| 設計意図 | 口頭伝達可 | ドキュメントに明示 |
| コロケーション | 推奨 | 強く推奨 |

> 📚 ファイルサイズの目安は、Robert C. Martin『Clean Code』の推奨(200行を標準、上限500行)に基づいています。

## 3. 従来経験の活かし方 [L2]

### 従来のシステム開発経験が活きる部分

```
┌─────────────────────────────────────────────────────────────────┐
│  従来の経験                    →    AI協働開発での適用          │
├─────────────────────────────────────────────────────────────────┤
│  構造体・データ構造設計        →    型定義、Value Object        │
│  モジュール分割                →    モジュール境界の設計        │
│  ヘッダーファイル              →    インターフェース定義        │
│  ビルドスクリプト(Makefile)  →    CI/CDパイプライン           │
│  テーブル設計・正規化          →    Entity設計、Aggregate境界   │
│  トランザクション設計          →    集約単位の整合性            │
└─────────────────────────────────────────────────────────────────┘
```

### 調整が必要な部分

```
┌─────────────────────────────────────────────────────────────────┐
│                    マインドセットの調整                          │
│                                                                 │
│   従来の考え方              →      AI協働での考え方             │
│                                                                 │
│   完璧になるまで            →      十分になったら               │
│   次に進まない                      進む                        │
│                                                                 │
│   一度決めたら              →      学習しながら                 │
│   変えない                          改善                        │
│                                                                 │
│   ドキュメントは            →      ドキュメントも               │
│   最初に完成                        進化                        │
│                                                                 │
│   手戻りは                  →      早期発見・早期修正           │
│   避けるべき                        (検証で担保)               │
└─────────────────────────────────────────────────────────────────┘
```

> 💡 従来の経験で培った「設計力」「判断力」はそのまま活用できる。
>    調整が必要なのは「いつ決めるか」「どこまで決めるか」のタイミング。
