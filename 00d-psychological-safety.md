## Level 1【環境】心理的安全性 — 失敗を恐れない環境 [L1]

> **階層での位置**: 環境層(L1) → 目的達成の前提条件
>
> **問いかけ**: この判断で、誰かが萎縮しないか?
>
> **関係**: 心理的安全性がなければ、丁寧にやる余裕(急がば回れ)が生まれない

### この層が防いでいる負の増幅:エラー隠蔽の増幅

増強のメカニズムは双方向に作用します（[増強の双方向性](./00b-augmentation.md)参照）。

L1が防いでいるのは**「エラー隠蔽の増幅」**という負の増幅ループです。

```
失敗を恐れる → エラーを隠す → AIが隠れたエラーの上に構築する
    ↑                                                        ↓
    └──────────── より大きな失敗 → さらに恐れる ←────────────┘
```

心理的安全性がない環境では、人間は小さなエラーを「なかったこと」にしがちです。
しかしAIは、隠されたエラーを前提として次の出力を生成します。
エラーは消えるのではなく、**AIの増幅力によって拡大して返ってきます。**

**心理的安全性は、エラーが増幅される前に表面化させるための環境設計です。**
「失敗を恐れない」は理想論ではなく、エラーの早期発見という実利的な目的を持っています。

### なぜ心理的安全性が重要なのか

「急がば回れ」が正しいとわかっていても、実践できないことがあります。

なぜか。

**失敗を恐れているからです。**

- 「検証で失敗したら、時間の無駄だったと思われる」
- 「損切りしたら、能力がないと思われる」
- 「タスク分解に時間をかけたら、遅いと思われる」

この恐れがある限り、人は「急いで」しまいます。
急いだ結果、手戻りが発生し、本当に遅くなる。

### 心理的安全性とは

心理的安全性とは、**失敗しても大丈夫だと思える環境**のことです。

```
┌─────────────────────────────────────────────────────────────────┐
│                    心理的安全性の効果                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  心理的安全性がない環境                                         │
│  ─────────────────────                                          │
│  失敗を恐れる → 検証を避ける → 問題が蓄積 → 大きな失敗          │
│       ↑                                                ↓        │
│       └────────────── さらに恐れる ←─────────────────┘          │
│                                                                 │
│  心理的安全性がある環境                                         │
│  ─────────────────────                                          │
│  失敗しても大丈夫 → 検証を頻繁に → 早期発見 → 小さな修正        │
│       ↑                                                ↓        │
│       └────────────── 自信がつく ←──────────────────┘           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### AI協働開発における心理的安全性

AI協働開発では、心理的安全性が特に重要です。

なぜなら、**失敗が学びの源泉**だからです。

| 従来の開発 | AI協働開発 |
|-----------|-----------|
| 失敗は避けるべきもの | 失敗は学びの機会 |
| 失敗したら隠したい | 失敗したらCLAUDE.mdに記録 |
| 失敗は恥ずかしい | 失敗は次の成功の種 |

### 心理的安全性の実践

```
┌─────────────────────────────────────────────────────────────────┐
│                心理的安全性を高める実践                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  損切りは敗北ではない                                           │
│  ───────────────────                                            │
│  3往復しても解決しないなら、撤退する。                          │
│  これは「諦め」ではなく「賢明な判断」。                         │
│  撤退後に人間が解決し、学びをCLAUDE.mdに記録すれば、            │
│  次回からは同じ問題で躓かない。                                 │
│                                                                 │
│  検証失敗は発見                                                 │
│  ──────────────                                                 │
│  typecheckやtestが失敗することは、悪いことではない。            │
│  問題を本番環境に持ち込む前に発見できた、ということ。           │
│  失敗を歓迎し、修正して次に進む。                               │
│                                                                 │
│  完璧を求めない                                                 │
│  ────────────                                                   │
│  最初から完璧なタスク分解はできない。                           │
│  まず分解して、レビューして、修正する。                         │
│  サイクルを回すことが、完璧を目指すより速い。                   │
│                                                                 │
│  失敗を責めない(自分も、他人も)                               │
│  ──────────────────────────────                                 │
│  失敗を責める文化は、失敗を隠す文化を生む。                     │
│  失敗を隠す文化は、学びを失う文化を生む。                       │
│  失敗を共有し、学びに変える文化を作る。                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 心理的安全性が確保されると

心理的安全性が確保されると、「急がば回れ」が自然と実践できるようになります。

- 失敗を恐れないから、検証を頻繁にできる
- 損切りを恥じないから、早期に撤退できる
- 完璧を求めないから、サイクルを回せる

**心理的安全性は、「急がば回れ」の土台です。**

### 意思決定バイアスの緩和 — AI過信を防ぐ [L2]

心理的安全性（L1）は、**意思決定バイアス**への対策も含みます。
特にAI協働開発では、AI特有のバイアスに注意が必要です。

#### AI協働開発で起きやすいバイアス

```
┌─────────────────────────────────────────────────────────────────┐
│              AI協働開発で注意すべきバイアス                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  オートメーションバイアス                                       │
│  ─────────────────────────                                      │
│  「AIが言っているから正しいはず」                               │
│  → AIの出力を無批判に受け入れてしまう                          │
│                                                                 │
│  確証バイアス                                                   │
│  ───────────                                                    │
│  「この方法で正しいはずだ」                                     │
│  → 自分の仮説を支持するAI出力だけを見てしまう                  │
│                                                                 │
│  サンクコストバイアス                                           │
│  ─────────────────                                              │
│  「ここまでやったから続けよう」                                 │
│  → 損切りすべきタイミングで継続してしまう                      │
│                                                                 │
│  アンカリングバイアス                                           │
│  ─────────────────                                              │
│  「最初にAIが提案したこの方法で」                               │
│  → 最初の提案に引きずられて他の選択肢を検討しない              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### バイアス緩和の実践

| バイアス | 緩和策 | 関連原則 |
|---------|--------|---------|
| **オートメーションバイアス** | 必ず検証を挟む | H1, S1「1文1検証」 |
| **確証バイアス** | 意図的に反証を求める | デビルズアドボケート |
| **サンクコストバイアス** | 損切りルールを機械的に適用 | P2「損切りは判断」 |
| **アンカリングバイアス** | 複数案を最初に出させる | F1「初期案生成はAIから」 |

#### デビルズアドボケート・プロトコル

特に重要な意思決定では、**意図的に反論を求める**プロトコルが有効です:

```
重要な設計判断
    │
    ↓
AIに「この設計の問題点を3つ挙げてください」と依頼
    │
    ↓
問題点を検討
    │
    ├─ 致命的な問題がある → 設計を見直す
    │
    └─ 許容できる → 問題点を記録して進める
```

**プロンプト例**:
```
この設計案について、批判的な視点から評価してください:
1. この設計の最大の弱点は何ですか?
2. この設計が失敗するとしたら、どんな理由が考えられますか?
3. 代替案として検討すべきアプローチはありますか?
```

> **R3「判断の可逆性」との関係**:
> バイアスに気づいたとき、判断を覆せる状態を維持することが重要です。

### フロー理論 — 集中状態を守る [L2]

心理的安全性（L1）は、**フロー状態**の維持にも関わります。

#### フロー状態とは

> **定義**: 活動に完全に没頭し、時間感覚を忘れるほど集中している状態
> （Mihaly Csikszentmihalyi, 1990）

```
┌─────────────────────────────────────────────────────────────────┐
│                  フロー状態の条件                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 明確な目標                                                  │
│     → 「1文1検証」で実現                                       │
│                                                                 │
│  2. 即座のフィードバック                                        │
│     → 検証ループで実現                                         │
│                                                                 │
│  3. スキルと難易度のバランス                                    │
│     → タスク分解で調整                                         │
│                                                                 │
│  4. 集中を妨げない環境                                          │
│     → 心理的安全性で実現                                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### AI協働開発でフローを維持する

| フロー条件 | AI協働での実現方法 |
|-----------|-------------------|
| 明確な目標 | 「1文1検証」でタスクを明確化 |
| 即座のフィードバック | 小さな検証サイクル |
| 適切な難易度 | AIで認知負荷を分散 |
| 集中維持 | 中断を最小化する設計 |

#### フローを妨げるもの

```
┌─────────────────────────────────────────────────────────────────┐
│                  フローを妨げる要因                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  外的要因                                                       │
│  ────────                                                       │
│  ・頻繁な割り込み                                               │
│  ・不明確な要件                                                 │
│  ・ツールの待ち時間                                             │
│                                                                 │
│  内的要因                                                       │
│  ────────                                                       │
│  ・失敗への恐れ → 心理的安全性で対処                           │
│  ・完璧主義 → P3「完璧より改善」で対処                         │
│  ・決断疲れ → タスク分解で対処                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> **心理的安全性との関係**:
> 失敗を恐れない環境があってこそ、フロー状態に入れます。

### 心理的安全性を高める具体的行動

心理的安全性は「気持ちの問題」ではなく、**具体的な行動の積み重ね**で構築されます。

#### 個人として実践すること

| 場面 | 具体的行動 | 言い換え(マインドセット) |
|------|-----------|--------------------------|
| エラーが出た時 | 「発見できた」と声に出す | 失敗 → 発見 |
| 3往復で解決しない時 | 損切りして記録する | 諦め → 賢明な判断 |
| わからない時 | 「わからない」と正直に認める | 無知 → 学習機会 |
| 他の人が失敗した時 | 「何を学んだ?」と聞く | 責める → 共に学ぶ |

```
┌─────────────────────────────────────────────────────────────────┐
│              個人の行動チェックリスト(日次)                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  □ 今日、検証失敗を「発見」と捉えられたか?                     │
│  □ 今日、損切り判断を躊躇なくできたか?                         │
│  □ 今日、学びをCLAUDE.mdに記録したか?                          │
│  □ 今日、「わからない」を正直に言えたか?                       │
│                                                                 │
│  → 4つ中3つ以上「はい」なら、心理的安全性を実践できている       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### チームとして実践すること

| 仕組み | 具体的アクション | 頻度 |
|--------|-----------------|------|
| 失敗共有会 | 「今週の失敗と学び」を共有(責めない) | 週1回 |
| 損切り称賛 | 損切りした人を「良い判断」と称える | 随時 |
| CLAUDE.md更新の評価 | 学びの記録を評価対象に含める | 評価時 |
| 新しい試みの許容 | 「試してダメならOK」を明文化 | 常時 |

#### チーム規範テンプレート

チームで心理的安全性を高めるために、以下の規範を明文化することを推奨します。

```
┌─────────────────────────────────────────────────────────────────┐
│                    チーム規範(テンプレート)                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 失敗は学びの機会である                                      │
│     - 失敗を責めない                                            │
│     - 失敗をCLAUDE.mdに記録することを奨励する                   │
│     - 「何を学んだか」を共有する文化を作る                      │
│                                                                 │
│  2. 損切りは賢明な判断である                                    │
│     - 3往復ルールを全員で守る                                   │
│     - 損切りした人を「良い判断」と称える                        │
│     - 撤退後の分析を重視する                                    │
│                                                                 │
│  3. 「わからない」は言っていい                                  │
│     - 質問を歓迎する                                            │
│     - 「聞いてくれてありがとう」と返す                          │
│     - 一人で抱え込まない                                        │
│                                                                 │
│  4. 完璧より改善を重視する                                      │
│     - 最初から完璧を求めない                                    │
│     - サイクルを回して改善する                                  │
│     - 小さな改善を積み重ねる                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
