## Level 4【実践】AIファースト — 人間が考える前にAIに初期案を [L2]

> **階層での位置**: 実践層(L4) → L0-L3を満たした上での積極活用
>
> **問いかけ**: AIの得意分野を活用しているか？人間だけで悩んでいないか？
>
> **関係**: L0-L3の「守り」の上に成り立つ「攻め」の実践

### なぜ「AIファースト」が必要なのか

現フレームワーク（v4.0まで）は、主に以下の観点で設計されていました:

- AIの弱点を補う（検証、シンプルさ）
- AIへの過度な依存を防ぐ（可逆性）
- AIとの協働リスクを管理する（心理的安全性）

これらは重要ですが、**AIの長所を積極的に活用する**視点が不足していました。

```
┌─────────────────────────────────────────────────────────────────┐
│            L0-L3 と L4 の関係                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  L0-L3: 守りの原則（必須）                                      │
│  ─────────────────────────                                      │
│  ・AIの弱点を補う                                               │
│  ・リスクを管理する                                             │
│  ・人間の制御を維持する                                         │
│  ・持続可能性を確保する                                         │
│                                                                 │
│  L4: 攻めの原則（推奨）                                         │
│  ─────────────────────                                          │
│  ・AIの長所を活用する                                           │
│  ・効率を最大化する                                             │
│  ・人間の認知資源を解放する                                     │
│  ・創造的な作業に集中させる                                     │
│                                                                 │
│  ★重要★                                                        │
│  L4はL0-L3のすべてを満たした上で適用される                      │
│  「攻め」は「守り」を壊してはならない                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### L4が最下層にある理由

L4の実践は、必ずL0-L3のすべてを満たした上で行われます。

| 違反パターン | 起きること | 正しいアプローチ |
|-------------|-----------|-----------------|
| L0違反のL4実践 | 短期的に速くても長期的に崩壊 | 持続可能性を確認してから効率化 |
| L1違反のL4実践 | AIへの過度な依存、人間の萎縮 | 心理的安全性を保ちながら委譲 |
| L2違反のL4実践 | 検証なしの丸投げ、品質崩壊 | 必ず検証を挟んでAI活用 |
| L3違反のL4実践 | 複雑な指示、AIの誤解 | シンプルな単位でAIに依頼 |

**L4を最下層に置くことで、「攻め」が「守り」を壊さない構造を保証します。**

## AIの長所：活用すべき5つの特性 [L2]

### 概要

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIの5つの長所                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ① パターン認識                                                 │
│     大量のコードベースで学習した設計パターンを即時認識          │
│     → 設計レビュー、リファクタリング提案、ベストプラクティス   │
│                                                                 │
│  ② 一貫性                                                       │
│     疲労による品質低下がない、24時間365日稼働可能               │
│     → スタイル統一、規約チェック、CI/CDでの継続監視            │
│                                                                 │
│  ③ 網羅性                                                       │
│     系統的なエッジケース列挙、人間の見落としを防止              │
│     → テストケース生成、境界条件分析、抜け漏れチェック         │
│                                                                 │
│  ④ 処理速度                                                     │
│     数千ファイルの同時分析、即座の応答                          │
│     → コードベース全体の分析、高速な検索・置換                 │
│                                                                 │
│  ⑤ 知識幅                                                       │
│     多言語・多フレームワークの知識を横断的に保持                │
│     → 技術選定、ベストプラクティス提案、言語間の翻訳           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 長所と短所の対比

AIの長所を活用する際は、同時に短所も認識しておく必要があります。

| 長所 | 対応する短所 | 人間の役割 |
|------|-------------|-----------|
| パターン認識 | 新規パターンの創造は苦手 | 創造的判断 |
| 一貫性 | 意図的な逸脱ができない | 例外判断 |
| 網羅性 | 優先度付けが苦手 | ビジネス判断 |
| 処理速度 | 深い理解は別問題 | 本質理解 |
| 知識幅 | 最新情報は持たない | 情報更新 |

## AIファースト原則 (F1-F4) [L2]

### F1: 初期案生成はAIから

```
┌─────────────────────────────────────────────────────────────────┐
│                    F1: 初期案生成はAIから                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  核心: 人間が白紙から考える前に、AIに初期案を生成させる         │
│                                                                 │
│  やること:                                                      │
│  ・設計の選択肢をAIに3-5案提示させる                           │
│  ・ボイラープレートコードはAIに生成させる                       │
│  ・テストケースの初期リストはAIに列挙させる                     │
│  ・ドキュメント草案はAIに書かせる                               │
│  ・エラーメッセージの文言はAIに提案させる                       │
│                                                                 │
│  やらないこと:                                                  │
│  ・AIの提案を無批判に採用する                                   │
│  ・人間の判断を省略する                                         │
│  ・L0-L3の原則を無視する                                        │
│  ・AIに「最終決定」を任せる                                     │
│                                                                 │
│  判断基準: 「これはAIが得意な作業か?」                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**具体例:**

```
【悪い例】人間が白紙から考える
人間: 「ログイン機能の設計を考えよう...」
→ 1時間悩む
→ やっと1案思いつく
→ Claudeに実装を依頼

【良い例】AIに初期案を依頼
人間: 「ログイン機能の設計案を3つ提示してください。
       それぞれのメリット・デメリットも説明してください。」
Claude: 「案1: セッションベース...、案2: JWT...、案3: OAuth...」
→ 5分で3案を比較検討
→ 人間が最適案を選択
→ 選択した案の実装を依頼
```

### F3: 網羅性チェックはAIに依頼

```
┌─────────────────────────────────────────────────────────────────┐
│                    F3: 網羅性チェックはAIに依頼                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  核心: 漏れ・抜けの検出はAIの網羅性を活用する                   │
│                                                                 │
│  やること:                                                      │
│  ・エッジケースの系統的列挙                                     │
│  ・テストケースの網羅性確認                                     │
│  ・エラーハンドリングの抜け漏れチェック                         │
│  ・APIパラメータの境界条件分析                                  │
│  ・セキュリティ観点のチェックリスト生成                         │
│                                                                 │
│  やらないこと:                                                  │
│  ・AIの列挙を「完全」と思い込む（人間も確認）                   │
│  ・ビジネス上の優先度判断をAIに任せる                           │
│  ・「どこまでカバーするか」の判断をAIに委ねる                   │
│                                                                 │
│  判断基準: 「見落としがあると致命的か?」                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**具体例: エッジケース列挙**

```
【指示例】
「以下の関数について、考慮すべきエッジケースを列挙してください:

function divide(a: number, b: number): number {
  return a / b;
}」

【AIの回答例】
1. b = 0 の場合（ゼロ除算）
2. a または b が NaN の場合
3. a または b が Infinity の場合
4. a または b が -Infinity の場合
5. 非常に大きな数値（オーバーフロー）
6. 非常に小さな数値（アンダーフロー）
7. 負数同士の除算
8. 正負混合の除算
9. a = 0 の場合（結果が0）
10. a = b の場合（結果が1）

→ 人間が見落としがちなケースをAIが系統的に列挙
```

### F4: プロンプト設計を最適化

```
┌─────────────────────────────────────────────────────────────────┐
│                    F4: プロンプト設計を最適化                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  核心: AIの能力を最大限引き出すプロンプト技法を適用する         │
│                                                                 │
│  状況別の技法選択:                                              │
│  ┌──────────────────┬────────────────────────────────────┐      │
│  │ 状況             │ 推奨技法                           │      │
│  ├──────────────────┼────────────────────────────────────┤      │
│  │ シンプルなタスク │ Zero-shot（そのまま指示）          │      │
│  │ 推論が必要       │ Chain of Thought（段階的思考）     │      │
│  │ 複数の選択肢     │ Tree of Thoughts（分岐探索）       │      │
│  │ 外部情報が必要   │ ReAct（推論+行動の交互実行）       │      │
│  │ 確実性が重要     │ Self-consistency（複数回+多数決）  │      │
│  └──────────────────┴────────────────────────────────────┘      │
│                                                                 │
│  判断基準: 「このタスクの複雑さ・重要度は?」                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**プロンプト技法の詳細:**

| 技法 | 説明 | 使用場面 |
|------|------|---------|
| **Zero-shot** | 例示なしで直接指示 | 明確で単純なタスク |
| **Few-shot** | 数例を示してから指示 | パターンを伝えたいとき |
| **Chain of Thought** | 「段階的に考えて」と指示 | 推論が必要なタスク |
| **Tree of Thoughts** | 複数の思考パスを探索 | 複雑な問題解決 |
| **ReAct** | 推論→行動→観察を繰り返し | 外部ツール連携時 |
| **Self-consistency** | 複数回実行して多数決 | 重要な判断 |

## 人間-AI認知労働分業 [L3]

### 分散認知の視点

AIを「認知的パートナー」として捉え、認知労働を最適に分業します。

```
┌─────────────────────────────────────────────────────────────────┐
│                人間-AI認知労働分業                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────┐          ┌─────────────┐                      │
│  │   人間      │◄────────►│    AI      │                      │
│  │             │  協働    │             │                      │
│  └──────┬──────┘          └──────┬──────┘                      │
│         │                         │                            │
│         ▼                         ▼                            │
│  ┌─────────────┐          ┌─────────────┐                      │
│  │ 判断・創造  │          │ 処理・生成  │                      │
│  │ 価値判断    │          │ パターン認識│                      │
│  │ 目標設定    │          │ 一貫性確保  │                      │
│  │ 例外処理    │          │ 網羅性チェック│                    │
│  │ 最終決定    │          │ 初期案生成  │                      │
│  └─────────────┘          └─────────────┘                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 分業表

| フェーズ | AIに委譲（長所活用） | 人間が担当（判断・創造） |
|---------|---------------------|------------------------|
| **設計** | 選択肢生成、パターン提案、トレードオフ分析 | 最終選択、価値判断、ビジネス整合性 |
| **実装** | ボイラープレート、テスト生成、定型コード | 複雑なロジック、状態管理設計、例外設計 |
| **レビュー** | 静的解析、スタイルチェック、網羅性確認 | セキュリティ判断、アーキテクチャ評価、コンテキスト判断 |
| **文書** | 初稿生成、API説明、コメント補完 | 設計意図、ビジネスコンテキスト、読者への配慮 |
| **デバッグ** | エラーパターン分析、修正案提示 | 根本原因分析、再発防止策、優先度判断 |
| **品質** | 24時間監視、一貫性チェック、自動テスト | 品質基準の設定、例外の許容判断 |

## よくある誤解と正しい理解 [L1]

| 誤解 | 正しい理解 |
|------|-----------|
| 「AIファースト = AIに丸投げ」 | AIに初期案を任せ、人間が判断・検証する |
| 「L4が最も重要」 | L4はL0-L3の上に成り立つ付加価値 |
| 「すべてAIに任せるべき」 | AIの得意分野に絞って委譲する |
| 「効率が最優先」 | 持続可能性(L0)が常に最優先 |
| 「AIが提案したら従うべき」 | 人間が批判的に評価し、最終判断する |
| 「AIファーストは手抜き」 | 人間の認知資源を創造的作業に集中させる戦略 |
